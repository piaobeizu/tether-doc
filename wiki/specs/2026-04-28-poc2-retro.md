## TL;DR

PoC-2 设计是 6 步 / 1 周硬验证。**实际跑了 5 步（step 4 connection migration 推迟）/ 半天**。webtransport-go v0.10 三方互通（Go server / Go client / Chrome browser 桌面+Android）全部 PASS；30 分钟长连接 0 leak；§11.T 风险点完全消除——**v0.1 不需要降级到 raw QUIC**。

主要 cost 不在 wire 协议本身（quic-go + webtransport-go 工作得很好），而在**真实部署环境的隐藏阻塞点**：
- 4 条 webtransport-go v0.10 没文档的非显然要求（P-01..P-04）
- Cert SAN 必含 dial host（W3C spec vs Chrome 实际行为差异，P-05）
- 用户系统代理拦截 WT 流量（最大坑，P-06）
- Android secure context 配置麻烦（Chrome flag 不稳，需 HTTPS tunnel）

## 什么做对了

### 1. "HARD GATE" 概念本身

把 PoC-2 标记为**实施前的硬验证关**——而不是"边写边踩"。这是关键决定。如果直接进 daemon 实现，那 6 个 P-XX gotcha 会分散在 1-2 周的实现期里，每个出现都打断 flow。集中在 PoC 阶段一次性踩完，**收益很大**。

### 2. Build tag + 独立 step 文件（继承 PoC-1 模式）

每个 step 一个 `.go` 文件 + 独立 build tag，互不干扰。新增 step 不破坏旧 step。

### 3. 写 `cert_dump.go` 诊断工具

step 3 卡住 cert 问题时，写了一个工具枚举 Chrome serverCertificateHashes 的全部 8 项要求并逐项打 ✓/✗。**确认 cert 100% 合规**那一刻，就把 debug 方向从"是不是 cert 问题"切成了"肯定不是 cert"——节省了大量盲猜时间。

### 4. Chrome netlog 抓诊断

`chrome://net-export/` + `netlog-viewer.appspot.com` 是发现 Clash 拦截的**唯一**有效手段。如果没这一步，会一直在 cert / firewall / quic-go 配置上瞎试。

### 5. quic-go + webtransport-go 选型本身

两个库工作可靠。30min 长连接 1MB heap 不变 / 0 panic / 0 reconnect。**§11.T 风险点消除**，v0.1 直接用，不需要 raw QUIC fallback。

## 什么意外炸了

### 🔥 P-06 系统代理：最大坑

**Clash Verge 系统代理拦截 WT 流量**，返回 `ERR_TUNNEL_CONNECTION_FAILED`。HTTP CONNECT 隧道**不支持 UDP/QUIC**。

- 影响：Chrome WT API 直接 fail，不到 server 一步
- 诊断难度：极高（cert / firewall / quic-go 全排查后才到这里）
- 真实部署：**绝大多数 dev 机器都装 Clash / Surge / V2Ray**——这是普遍问题

**这条没在 v0.1 spec 里预料到**。`§11.K Connectivity` 文档现已加警告，但部署用户必须记得给 server IP 加 DIRECT 规则。

**lesson**：未来 PoC 真机测试时，第一步先**禁用 / bypass 系统代理**——而不是当成最后一个怀疑对象。

### P-05 cert SAN：W3C spec vs Chrome 行为差异

W3C WebTransport spec 暗示 `serverCertificateHashes` 替代了完整 cert 验证。**Chrome 实际仍执行 RFC 6125 hostname verification**。cert SAN 必须包含 dial host。

- 影响：仅 `127.0.0.1` 的 SAN → 跨公网 dial 公网 IP 直接拒
- 诊断：写 `cert_dump.go` 工具发现 cert 全合规后，反推这是 Chrome 实现细节
- 修复：cert 生成时通过 `TETHER_POC_EXTRA_IP` env 注入

**lesson**：浏览器 WT API 的 spec ≠ 实际行为，**必须实测验证**。生产 cert 必须含真实 dial host 的所有 IP/DNS。

### P-01..P-04 webtransport-go v0.10 没文档要求

```
P-01 webtransport.ConfigureHTTP3Server(h3) 必显式调（否则 client "didn't enable WebTransport"）
P-02 EnableStreamResetPartialDelivery: true server + client 都要开
P-03 Datagram 双层都要开（http3.EnableDatagrams + quic.EnableDatagrams）
P-04 client tls.NextProtos 必含 "h3"
```

每条都不在 README / godoc 里直接写明，靠 trial-and-error 找。每个 ~5 分钟 fix，但累积起来 stack。

**lesson**：选型新 lib 时，"happy path 例子能跑" ≠ "lib 行为完全清楚"。生产前最好把所有 config 项都解开看清楚。

### Android secure context 比 desktop 麻烦

桌面 Chrome 用 `chrome://flags#unsafely-treat-insecure-origin-as-secure` 加白名单很顺。**Android Chrome 同 flag 不稳**。最终用 Cloudflare quick tunnel 给 HTML 加 HTTPS（直接绕开 secure context 问题）。

**lesson**：Tauri Mobile 集成时**HTML 必须从 Tauri WebView 内部加载**（file://, secure context 自动满足）——不会有 Android Chrome 这个 friction。**Tauri WebView 的 secure context 比 Android Chrome 反而更稳**。

## 对 v0.1 实施阶段的影响

### 必须做的（基于 PoC-2 发现）

1. **`tether doctor` 命令加"系统代理检测"**（§11.R 已锁但具体内容要包含）：
   - 检测 `HTTPS_PROXY` / `HTTP_PROXY` env
   - 检测 macOS / Windows 系统级 proxy 配置
   - 探测 outbound UDP/443 是否能到 server（不被代理拦）
   - 输出"建议在 Clash/Surge 配置 IP-CIDR DIRECT 规则"指引

2. **Cert 生成代码的 SAN**（D-12 配套实现）：必须把 server 实际部署的所有 IP/DNS 都加进 cert SAN。Tether 配对协议要把 server 公开地址告诉 CLI/App。

3. **`tether server` 启动 banner 打印 fingerprint + SAN**：让用户能立即验证 cert 跟 dial host 匹配。

4. **部署文档（§11.K）proxy 警告**：✅ 已加。但实施期还要写一段"完整设置 walkthrough"覆盖 Clash 配置具体步骤。

### 推荐做的（不阻塞但提升体验）

5. **PoC-2 的 6 个 step 转 CI 集成测试**（§11.Q P0）：把 step1-6 做成 `make poc2-ci`，未来 webtransport-go 升级 / quic-go 升级时自动跑回归。

6. **`step3_browser/index.html` 留作 dogfood 工具**：实施期任何时候想验"server 还能不能连"，浏览器开 HTML 30 秒得答案。

7. **真机 step 4 connection migration 在 dogfood 期顺带验**：通勤切 4G/WiFi 是天然测试场景。不用单独 PoC。

### 不阻塞但要注意

8. **模拟丢包未测**（`tc qdisc add ... netem loss N%`）：localhost 实测 0% loss 不能直接外推到 4G。dogfood 期天然会暴露真实丢包行为。

9. **多小时 / 多天稳定性未测**：30min ≠ 实际通勤+办公一整天的连续运行。dogfood 期会自动覆盖。

## 没 capture 但值得记的元 lessons

### 1. PoC 的"真实环境验证"必须真用真实环境

Localhost happy path 容易过。**生产是 GCP VM + 跨公网 + 用户真实代理 + 真实手机**。step 1/2/5/6 全在 localhost 跑 PASS，看似 hard gate 通过；step 3 跨公网才暴露 P-05 / P-06 / Android secure context 三个真实环境问题。

**lesson**：未来 PoC 设计时，**至少有一步必须跨公网 + 真实客户端环境**，否则不算覆盖。

### 2. 诊断工具的投入回报极高

写 `cert_dump.go` 用了 ~10 分钟，**节省了大约 1 小时的盲猜**。Chrome netlog 用了 ~5 分钟（学怎么打开），**节省了大约 2 小时**（如果没看 netlog，会一直在 cert/firewall 上瞎试）。

**lesson**：调试卡 30 分钟以上时，**优先停下来写诊断工具 / 抓 trace**，不要继续盲猜。

### 3. 用户场景里"系统代理"是一类必须独立思考的环境因素

Clash / Surge / V2Ray / Lantern / WARP / 各种 VPN 全是这类。它们对**TCP/HTTP** 用户透明，对 **UDP/QUIC/WebTransport** 直接拒。任何依赖 QUIC 的产品都必须考虑。

**lesson**：tether 的 onboarding 流程（pair 时）应该自动检测系统代理 + 给用户写好 bypass 指令。这是真实可用性细节。

### 4. webtransport-go 不是 zero-friction

库选型时假设"webtransport-go 是 quic-go 官方封装，应该 just works"。**实际有 4 条没文档的非显然要求**。

**lesson**：未来引入新依赖前，**至少跑一个端到端 example**——并把所有 config 选项 grep 一遍，看哪些是必需的。
