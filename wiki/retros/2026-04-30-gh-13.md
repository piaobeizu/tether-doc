## 1. Assumptions

**Held**:

- **stream-json data plane is stable across CC v2.1.x** — B' verified across 4 patches (.120 → .123); system/init keys byte-identical; A.2 graceful-degrade absorbed the one variation (extra `user` envelope on .120). Strong forward-compat for the parse layer.
- **`claude` is a native ELF binary** — confirmed via `/proc/<pid>/maps`; no Node embedding needed (F.1).
- **kill+resume is safe** (no double-execution; task intent lost) — C station scenario probe confirmed empirically. `claude --resume <sid>` reads jsonl on disk, model treats unanswered tool_use as "no result", proceeds without retry.
- **cwd-bucketing forces explicit policy** — `~/.claude/projects/<encoded-cwd>/<sid>.jsonl` lookup on `--resume` made E.5 a hard-gate decision; chose strategy (a) fixed `$HOME` for v0.1.

**Did not hold**:

- **"system/init fires on spawn"** — initial assumption was that cc emits init immediately after process start. **Discovered late in Step 5**: in `--input-format stream-json` mode, cc waits for the FIRST user envelope on stdin before init fires. Forced API split: `New()` returns immediately, `Start(ctx, initialMessage)` sends + waits. Documented as a non-obvious contract in `New` godoc. Cost: ~30 min of debugging + one API redesign.
- **"happy-cli's fd 3 thinking-state side channel is part of cc's contract"** — initial reading of happy's source led me to call this a "happy private invention". Code review mid-investigation flipped it to "actually CC's undocumented diagnostic fd". Either way: not in our public-contract surface; A.1 dropped fd 3 entirely.

## 2. Surprises

- **30 messages of design before any code felt long, but it kept being load-bearing**. Each crux station (A→G) surfaced at least one finding the handoff had glossed over. The cwd-gap discovered mid-discussion (E.5) was a real bug-class issue that would have taken hours to debug post-implementation.
- **Round-2 code review on my own implementation found 13 findings** — 7 of them race / panic / leak class. Solo coding without a second reviewer leaves real bugs in the tree even with TDD discipline. Multi-agent review (concurrency-focused subagent + error-path/coverage-focused subagent) caught all 7 race-class issues in a single pass.
- **Sonnet was CHEAPER than Haiku for the C' cost test** — $0.051 vs $0.111 for the same prompt. Counter-intuitive: Haiku had 74k cache_creation tokens because the gstack SessionStart hooks dumped the full skill catalog into context; Sonnet's request landed differently in cache. The per-token rate intuition was wrong; cache composition dominated.
- **Code review found a TEST-infrastructure race**, not a source race: `bytes.Buffer` shared across goroutines in `control_test.go` is not goroutine-safe. The review's `-race` insistence caught it; without it the test suite "passed" while still racing.
- **`exec.Cmd.Wait()` race** required `WaitOnce` helper — was preempted by the first review (S1 #2) before it could bite. Subprocess teardown patterns under race conditions are easy to get wrong.

## 4. Reusability — patterns worth carrying forward

- **Envelope tagged-union + DecodeX-on-demand** (`message.go` pattern): minimal common shape (Type, Subtype, Raw bytes), specific decoders called by consumers, unknown types pass through. Generalizes to any vendor-CLI integration where wire schema may evolve.
- **`Session.Recover()` as unified primitive** (`session.go`): one method covers plugin-reload / pod-restart / daemon-crash / WT-reconnect. State machine + safe-to-reload gate + stale-state override + AbortPending (close pending control map) — the four constraints are independent of cc and apply to any backend abstraction.
- **Spike driver pattern** (`cmd/spike`): scenario IDs + per-scenario PASS/FAIL/SKIP + markdown report + cost characterization in one binary. Reusable for any lib package where contract verification needs end-to-end runs against a real backend.
- **Static contract guards** (`contracts_test.go`): negative-contract verification via source-file scan (E.1 forbids jsonl writes; A.1 forbids fd 3 wiring). Cheaper than runtime tests for "absence of X" properties.
- **Multi-agent review (concurrency-focused + error-focused subagents in parallel)** for solo-developer code review. Cost of one pass ~5-10 min, finds real bugs in race / leak / coverage dimensions that single-reviewer self-review misses.

## 5. Would do differently

- **Run scenario 1 (full conversation) BEFORE writing parser/control glue**. The system/init-after-first-Send discovery would have surfaced as an integration test failure on day 1 instead of week 1. The "build the smallest end-to-end happy path first, then layer subsystems" rule applied better than "build subsystems then integrate".
- **Run race detector (`go test -race`) in every commit, not just at review time**. Would have caught C2 (mu-protected reads of s.sub / s.control) at Step 5 instead of after merge-prep. Adding `-race` to standard `go test` invocation is essentially free.
- **Use `.progress.md` to log decisions and dead-ends as we go**. We didn't, and the retro had to reconstruct context from the conversation log. For future tickets this would also feed `/polyforge:debug` if anything went sideways.
- **Spike driver could be auto-runnable from CI** — currently manual `go run ./cmd/spike`; if it generated `.attestations/` JSON in the format `/polyforge:commit M2` expects, this ticket wouldn't have needed `--skip-m2` reasoning. Minor.
